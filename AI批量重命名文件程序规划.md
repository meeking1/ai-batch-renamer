# AI 批量重命名文件程序规划

## 1. 项目目标

开发一款可在 Windows 7、Windows 10、Windows 11 上运行的桌面程序，帮助用户通过自然语言批量重命名文件。

用户可以选择一批文件，输入重命名规则或多个目标名称，程序先生成重命名预览，用户确认后再执行实际改名。核心目标是让非技术用户也能用自然语言完成复杂、批量、可回滚的文件命名工作。

## 2. 目标用户

- 经常整理照片、视频、文档、扫描件、素材文件的普通用户
- 行政、财务、设计、摄影、电商运营等需要批量处理文件名的办公用户
- 需要按固定规则归档文件的企业内部用户
- 不熟悉正则表达式或批处理脚本，但希望快速完成批量命名的人

## 3. 核心使用场景

### 3.1 自然语言规则重命名

用户输入：

> 把这些照片按拍摄时间命名为“2026年春节旅行-001、002、003”

程序生成：

- IMG_3821.JPG -> 2026年春节旅行-001.JPG
- IMG_3822.JPG -> 2026年春节旅行-002.JPG
- IMG_3823.JPG -> 2026年春节旅行-003.JPG

### 3.2 多个名字逐个替换

用户输入或粘贴多个名称：

```text
合同-北京客户
合同-上海客户
合同-广州客户
```

程序按文件当前排序逐个匹配：

- 001.pdf -> 合同-北京客户.pdf
- 002.pdf -> 合同-上海客户.pdf
- 003.pdf -> 合同-广州客户.pdf

### 3.3 查找替换

用户输入：

> 把文件名里的“草稿”替换成“正式版”，保留原来的编号

程序生成：

- 产品说明-草稿-v1.docx -> 产品说明-正式版-v1.docx

### 3.4 提取文件信息命名

支持从文件属性中提取信息：

- 图片 EXIF 拍摄时间
- 文件创建时间
- 文件修改时间
- 原始文件名
- 文件扩展名
- 文件夹名

示例：

> 用修改日期加原文件名重命名，格式为“YYYY-MM-DD 原文件名”

### 3.5 撤销上一次重命名

用户误操作后，可以一键撤销最近一次批量重命名。

## 4. 功能范围

### 4.1 MVP 版本功能

第一版建议只做高价值、低风险功能：

1. 选择文件或拖拽文件到窗口
2. 显示文件列表：原文件名、预览新文件名、状态
3. 输入自然语言重命名要求
4. 输入多个目标名称并按顺序替换
5. 保留原扩展名
6. 重命名前自动检查冲突、非法字符、空文件名、重复文件名
7. 执行前必须显示预览
8. 执行后生成操作记录
9. 支持撤销最近一次操作
10. 支持 Windows 7、10、11 安装和运行

### 4.2 后续增强功能

- 文件夹批量重命名
- 按 EXIF、音视频元数据、PDF 标题、Office 文档属性命名
- 正则表达式高级模式
- 保存常用命名模板
- 文件排序规则：名称、时间、大小、手动拖动排序
- 支持递归处理子文件夹
- 支持离线小模型或本地规则引擎
- 企业版支持批量任务日志、权限控制、审计记录

## 5. Windows 兼容策略

### 5.1 推荐技术路线

推荐使用 C# + WPF + .NET Framework 4.8。

原因：

- Windows 7 SP1、Windows 10、Windows 11 均可支持
- WPF 适合做传统 Windows 桌面软件
- .NET Framework 4.8 在 Windows 生态稳定成熟
- 文件系统操作、安装包、桌面集成较方便
- 相比 Electron，程序体积更小，对老电脑更友好

注意：

- Windows 7 需要 SP1，并安装 .NET Framework 4.8 运行时
- TLS 1.2 在 Windows 7 上可能需要系统补丁或程序内兼容处理
- 如果要调用云端 AI API，必须特别测试 Windows 7 的 HTTPS 兼容性

### 5.2 可选技术路线

#### 方案 A：C# WPF + .NET Framework 4.8

推荐用于正式产品。

优点：

- Windows 兼容性好
- 安装包体积可控
- 原生桌面体验好
- 适合文件系统工具

缺点：

- UI 现代感需要额外设计
- 跨平台能力弱

#### 方案 B：Electron + Node.js

适合快速做界面原型，但不推荐作为 Win7 兼容优先方案。

优点：

- UI 开发快
- 前端人才容易参与
- 生态丰富

缺点：

- 新版 Electron 不支持 Windows 7
- 旧版 Electron 有安全和维护风险
- 安装包体积较大

#### 方案 C：Qt C++ / PyQt

适合有 C++ 或 Python 桌面开发经验的团队。

优点：

- 跨平台能力强
- 性能好

缺点：

- 开发和打包复杂度更高
- AI/API、安装包、自动更新需要更多工程处理

## 6. 产品流程设计

### 6.1 主流程

1. 用户拖入文件或点击选择文件
2. 程序读取文件列表
3. 用户选择模式：
   - 自然语言规则
   - 多名称替换
   - 查找替换
   - 模板命名
4. 用户输入需求
5. 程序生成新文件名预览
6. 程序执行安全检查
7. 用户确认
8. 程序批量重命名
9. 程序记录原名和新名
10. 用户可撤销

### 6.2 重命名前安全检查

必须检查：

- 新文件名不能为空
- 新文件名不能包含 Windows 非法字符：`< > : " / \ | ? *`
- 同一目录下不能出现重复文件名
- 不能覆盖已有文件，除非用户明确选择覆盖策略
- 文件是否正在被占用
- 路径长度是否超过系统限制
- 是否保留扩展名
- 是否存在大小写变化导致的文件系统冲突

### 6.3 操作记录

每次执行重命名后保存一份 JSON 日志：

```json
{
  "operationId": "20260701-153000",
  "createdAt": "2026-07-01T15:30:00+08:00",
  "items": [
    {
      "oldPath": "D:\\Files\\001.pdf",
      "newPath": "D:\\Files\\合同-北京客户.pdf",
      "status": "success"
    }
  ]
}
```

日志用途：

- 撤销上一次操作
- 排查失败原因
- 给用户提供操作凭证

## 7. AI 能力设计

### 7.1 AI 的职责边界

AI 只负责生成“建议的新文件名”，不直接操作文件系统。

实际文件修改必须由本地程序完成，并经过：

1. 本地规则校验
2. 冲突检测
3. 用户确认

这样可以降低误操作风险。

### 7.2 AI 输入

发送给 AI 的信息建议包括：

- 用户自然语言要求
- 文件原始名称列表
- 文件扩展名
- 文件排序
- 可选文件元数据
- 输出格式要求

示例：

```json
{
  "instruction": "按合同客户城市命名，保留编号",
  "files": [
    { "index": 1, "name": "scan001.pdf", "extension": ".pdf" },
    { "index": 2, "name": "scan002.pdf", "extension": ".pdf" }
  ],
  "rules": {
    "keepExtension": true,
    "noPathSeparators": true,
    "outputJsonOnly": true
  }
}
```

### 7.3 AI 输出

要求 AI 返回结构化 JSON，而不是普通文本：

```json
{
  "items": [
    { "index": 1, "newBaseName": "合同-北京客户" },
    { "index": 2, "newBaseName": "合同-上海客户" }
  ]
}
```

程序再把 `newBaseName` 和原扩展名组合成最终文件名。

### 7.4 AI 失败兜底

需要处理：

- 网络不可用
- API 超时
- 返回格式错误
- 返回数量和文件数量不一致
- 返回非法字符
- 返回重复名称
- 用户需求不明确

兜底策略：

- 明确提示错误原因
- 允许用户重新生成
- 保留上一次有效预览
- 支持不用 AI 的本地规则模式

## 8. 多名称替换规则

这是一个非常实用的功能，应在 MVP 中支持。

### 8.1 输入方式

用户可以在文本框中按行输入多个名称：

```text
张三身份证
李四身份证
王五身份证
```

每一行对应一个文件。

### 8.2 匹配规则

默认按当前文件列表排序逐个匹配：

- 第 1 行名称 -> 第 1 个文件
- 第 2 行名称 -> 第 2 个文件
- 第 3 行名称 -> 第 3 个文件

### 8.3 数量不一致处理

- 名称数量少于文件数量：提示缺少名称，不允许直接执行
- 名称数量多于文件数量：提示多余名称，可忽略或返回修改
- 空行默认忽略，也可提供“保留空行为跳过文件”的高级选项

## 9. 界面规划

### 9.1 主窗口区域

建议布局：

- 顶部工具栏：添加文件、添加文件夹、清空、撤销
- 左侧或上方：重命名模式选择
- 中部：文件表格
- 右侧或下方：输入区和生成按钮
- 底部：状态栏和执行按钮

### 9.2 文件表格字段

| 字段 | 说明 |
|------|------|
| 序号 | 当前排序 |
| 原文件名 | 当前文件名 |
| 新文件名预览 | 即将改成的名称 |
| 状态 | 正常、重复、非法、占用、失败 |
| 路径 | 可折叠显示完整路径 |

### 9.3 关键交互

- 支持拖拽文件进入窗口
- 支持表格内手动调整排序
- 支持双击编辑单个预览名称
- 执行按钮默认禁用，只有所有检查通过后才可点击
- 重命名前弹出确认框
- 重命名后显示成功、失败、跳过数量

## 10. 数据与配置

### 10.1 本地配置

保存内容：

- API 地址
- API Key，建议使用 Windows 凭据管理器或加密存储
- 默认是否保留扩展名
- 默认排序方式
- 最近打开目录
- 常用命名模板

### 10.2 敏感信息

不要把 API Key 明文写入配置文件。

推荐：

- Windows Credential Manager
- DPAPI 加密
- 企业部署时由管理员注入环境变量或配置

## 11. 安全与隐私

### 11.1 文件内容默认不上传

MVP 阶段建议只上传文件名和必要元数据，不上传文件内容。

原因：

- 降低隐私风险
- 降低成本
- 提高速度
- 避免企业用户顾虑

### 11.2 用户提示

在设置页明确说明：

- 是否使用云端 AI
- 会发送哪些数据
- 文件内容是否上传
- API Key 存储方式

### 11.3 本地优先原则

能在本地完成的规则不要调用 AI，例如：

- 添加前缀
- 添加编号
- 查找替换
- 大小写转换
- 按日期模板命名

### 12.3 关键服务

#### RenamePreviewService

负责根据用户输入生成预览，不直接改文件。

#### RenameValidationService

负责检查新名称是否合法。

#### RenameExecutionService

负责执行真实文件重命名，并写入操作日志。

#### UndoService

负责读取最近一次日志并反向重命名。

#### AiNamingService

负责调用 AI，返回结构化命名建议。

## 16. 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| AI 返回不稳定 | 文件名错误 | 使用 JSON Schema、本地校验、预览确认 |
| 用户误操作 | 文件混乱 | 强制预览、操作日志、一键撤销 |
| Windows 7 HTTPS 兼容 | API 调用失败 | 测试 TLS 1.2，提供系统要求说明 |
| 文件名冲突 | 重命名失败或覆盖 | 执行前完整冲突检测 |
| 文件正在被占用 | 部分失败 | 单文件失败不影响整体，清晰提示 |
| 隐私顾虑 | 用户不敢使用 | 默认不上传文件内容，明确隐私说明 |
| API Key 泄露 | 安全风险 | 使用凭据管理器或 DPAPI |

## 17. MVP 验收标准

MVP 完成时应满足：

1. 用户可以添加至少 100 个文件并稳定预览
2. 可以通过多行名称批量替换文件名
3. 可以通过自然语言生成文件名建议
4. 所有重命名操作执行前都有预览
5. 非法名称、重复名称、冲突名称会阻止执行
6. 重命名后可以撤销最近一次操作
7. 中文文件名、中文路径可正常处理
8. Windows 7 SP1、Windows 10、Windows 11 均可运行
9. API Key 不明文存储
10. 程序崩溃不会导致无日志的半完成状态

## 18. 建议的第一版界面文案

主输入框占位文案：

> 描述你想怎样重命名这些文件，例如：按日期命名为“项目资料-001、002、003”

多名称模式说明：

> 每行一个新名称，将按当前文件顺序依次匹配。默认保留原扩展名。

执行按钮：

> 确认重命名

撤销按钮：

> 撤销上次操作

错误提示示例：

> 有 2 个文件的新名称重复，请修改后再执行。

## 19. 下一步建议

建议先从 MVP 原型开始，不要一开始就做过多 AI 能力。

优先实现顺序：

1. 文件列表和拖拽
2. 多名称替换
3. 预览和校验
4. 执行重命名
5. 撤销
6. 接入 AI 命名
7. 打包安装

这样可以先把“批量重命名工具”做稳，再让 AI 成为增强能力，而不是让整个产品依赖 AI 的不确定性。
